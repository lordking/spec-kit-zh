## 清单目的："英语单元测试"

**关键概念**：清单是**需求编写的单元测试** - 它在给定领域验证需求的质量、清晰度和完整性。

**不用于验证/测试**：

- ❌ 不是"验证按钮正确点击"
- ❌ 不是"测试错误处理工作"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否与规格匹配

**用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层次需求？"（完整性）
- ✅ "是否用具体尺寸/定位量化了'显著显示'？"（清晰性）
- ✅ "所有交互元素的悬停状态需求是否一致？"（一致性）
- ✅ "是否为键盘导航定义了可访问性需求？"（覆盖率）
- ✅ "当徽标图像加载失败时，规格是否定义了会发生什么？"（边缘情况）

**比喻**：如果你的规格是用英语编写的代码，清单就是它的单元测试套件。你在测试需求是否编写良好、完整、无歧义，并准备好实施 - 而不是测试实现是否工作。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你必须考虑用户输入（如果不为空）。

## 执行步骤

1. **设置**：从仓库根目录运行 `{SCRIPT}` 并解析 JSON 以获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须是绝对的。
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **澄清意图（动态）**：推导最多三个初始上下文澄清问题（没有预制的目录）。它们必须：
   - 根据用户的措辞 + 从 spec/plan/tasks 提取的信号生成
   - 只询问实质上改变清单内容的信息
   - 如果已在 `$ARGUMENTS` 中明确，则单独跳过
   - 优先精确而非广度

   生成算法：
   1. 提取信号：功能领域关键词（例如，auth、延迟、UX、API）、风险指标（"关键"、"必须"、"合规"）、利益相关者提示（"QA"、"审查"、"安全团队"）和显式可交付成果（"a11y"、"回滚"、"契约"）。
   2. 将信号聚类为候选重点领域（最多 4 个）并按相关性排序。
   3. 识别可能的受众和时间（作者、审查者、QA、发布）如果未明确。
   4. 检测缺失维度：范围广度、深度/严格度、风险重点、排除边界、可衡量的验收标准。
   5. 从以下原型制定问题：
      - 范围细化（例如，"这应该包括与 X 和 Y 的集成接触点，还是仅限于本地模块正确性？"）
      - 风险优先级（例如，"这些潜在风险领域中哪些应接收强制性门禁检查？"）
      - 深度校准（例如，"这是轻量级提交前合理性列表还是正式发布门禁？"）
      - 受众框架（例如，"这将仅由作者使用还是同行在 PR 审查期间使用？"）
      - 边界排除（例如，"我们是否应明确排除本轮的性能调整项目？"）
      - 场景类别差距（例如，"未检测到恢复流程 - 回滚/部分失败路径是否在范围内？"）

   问题格式化规则：
   - 如果呈现选项，生成一个紧凑的表格，列：选项 | 候选 | 为什么重要
   - 限制为最多 A–E 选项；如果自由形式的答案更清晰，则省略表格
   - 永远不要求用户重述他们已说过的话
   - 避免推测性类别（无虚构）。如果不确定，明确询问："确认 X 是否属于范围。"

   当交互不可能时的默认值：
   - 深度：标准
   - 受众：如果与代码相关则为审查者（PR）；否则为作者
   - 重点：前 2 个相关性聚类

   输出问题（标签 Q1/Q2/Q3）。回答后：如果仍有 ≥2 个场景类别（备用/异常/恢复/非功能领域）不清楚，你**可**提出最多两个针对性后续问题（Q4/Q5），每个问题附带一行理由（例如，"未解决的恢复路径风险"）。总问题数不要超过五个。如果用户明确拒绝更多，则跳过升级。

3. **理解用户请求**：合并 `$ARGUMENTS` + 澄清答案：
   - 推导清单主题（例如，安全、审查、部署、ux）
   - 整合用户提到的显式必须包含的项目
   - 将焦点选择映射到类别支架
   - 从 spec/plan/tasks 推断任何缺失的上下文（不要虚构）

4. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围
   - plan.md（如果存在）：技术细节、依赖项
   - tasks.md（如果存在）：实现任务

   **上下文加载策略**：
   - 仅加载与活跃重点领域相关的必要部分（避免完整文件转储）
   - 优先将长部分总结为简洁的场景/需求项目
   - 使用渐进式披露：仅在检测到差距时添加后续检索
   - 如果源文档很大，生成临时摘要项目而不是嵌入原始文本

5. **生成清单** - 创建"需求的单元测试"：
   - 如果 `FEATURE_DIR/checklists/` 目录不存在，则创建它
   - 生成唯一的清单文件名：
     - 使用基于领域的简短、描述性名称（例如，`ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
     - 如果文件存在，追加到现有文件
   - 从 CHK001 开始顺序编号项目
   - 每次 `/speckit.checklist` 运行创建一个新文件（绝不覆盖现有清单）

   **核心原则 - 测试需求，而不是实现**：
   每个清单项目必须评估**需求本身**：
   - **完整性**：所有必需的需求都存在吗？
   - **清晰性**：需求无歧义且具体吗？
   - **一致性**：需求彼此一致吗？
   - **可衡量性**：可以客观验证需求吗？
   - **覆盖率**：所有场景/边缘情况都解决了吗？

   **类别结构** - 按需求质量维度分组项目：
   - **需求完整性**（是否记录了所有必需的需求？）
   - **需求清晰性**（需求具体且无歧义吗？）
   - **需求一致性**（需求无冲突地一致吗？）
   - **验收标准质量**（成功标准可衡量吗？）
   - **场景覆盖率**（所有流程/情况都解决了吗？）
   - **边缘情况覆盖率**（边界条件定义了吗？）
   - **非功能需求**（性能、安全、可访问性等 - 它们指定了吗？）
   - **依赖项和假设**（它们记录和验证了吗？）
   - **歧义和冲突**（需要澄清什么？）

   **如何编写清单项目 - "英语单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证着陆页显示 3 个剧集卡片"
   - "测试桌面上的悬停状态是否正确工作"
   - "确认徽标点击导航到主页"

   ✅ **正确**（测试需求质量）：
   - "是否用可衡量标准定义了视觉层次需求？"[清晰性]
   - "UI 元素的确切数量和定位是否明确指定？"[完整性]
   - "所有交互元素的交互状态需求是否一致定义？"[一致性]
   - "所有交互 UI 是否定义了键盘导航需求？"[覆盖率]
   - "图像加载失败时是否定义了回退行为？"[边缘情况]
   - "异步剧集数据的加载状态是否定义？"[完整性]
   - "竞争 UI 元素的视觉层次是否定义？"[清晰性]

   **项目结构**：
   每个项目应遵循此模式：
   - 询问需求质量的问题格式
   - 专注于编写的内容（或未编写）在 spec/plan 中
   - 在括号中包含质量维度 [完整性/清晰性/一致性等]
   - 检查现有需求时引用规格部分 `[Spec §X.Y]`
   - 检查缺失需求时使用 `[Gap]` 标记

   **按质量维度的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理需求？"[Gap]
   - "是否为所有交互元素指定了可访问性需求？"[完整性]
   - "是否为响应式布局定义了移动断点需求？"[Gap]

   清晰性：
   - "是否用特定时序阈值量化了'快速加载'？"[清晰性, Spec §NFR-2]
   - "是否明确后pub了'相关剧集'的选择标准？"[清晰性, Spec §FR-5]
   - "是否用可衡量的视觉属性定义了'显著'？"[歧义, Spec §FR-4]

   一致性：
   - "所有页面的导航需求是否一致？"[一致性, Spec §FR-10]
   - "着陆页和详情页之间的卡片组件需求是否一致？"[一致性]

   覆盖率：
   - "是否为零状态场景（无剧集）定义了需求？"[覆盖率, 边缘情况]
   - "是否解决了并发用户交互场景？"[覆盖率, Gap]
   - "是否为部分数据加载失败指定了需求？"[覆盖率, 异常流程]

   可衡量性：
   - "视觉层次需求是否可衡量/可测试？"[验收标准, Spec §FR-1]
   - "能否客观验证'平衡的视觉权重'？"[可衡量性, Spec §FR-2]

   **场景分类和覆盖率**（需求质量焦点）：
   - 检查以下需求是否存在：主要、备用、异常/错误、恢复、非功能场景
   - 对每个场景类别，询问："[场景类型]需求是否完整、清晰且一致？"
   - 如果场景类别缺失："[场景类型]需求是故意被排除还是缺失？"[Gap]
   - 在发生状态突变时包括弹性/回滚："是否为迁移失败定义了回滚需求？"[Gap]

   **可追溯性要求**：
   - 最低限度：≥80%的项目必须包含至少一个可追溯性引用
   - 每个项目应引用：规格部分 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 如果不存在 ID 系统："是否建立了需求和验收标准 ID 方案？"[可追溯性]

   **呈现和解决问题**（需求质量问题）：
   询问有关需求本身的问题：
   - 歧义："是否用具体指标量化了术语'快速'？"[歧义, Spec §NFR-1]
   - 冲突："§FR-10 和 §FR-10a 之间的导航需求是否冲突？"[冲突]
   - 假设："是否验证了'始终可用的 podcast API'的假设？"[假设]
   - 依赖项："是否记录了外部 podcast API 需求？"[依赖项, Gap]
   - 缺失定义："是否用可衡量标准定义了'视觉层次'?"[Gap]

   **内容整合**：
   - 软上限：如果原始候选项目 > 40，按风险/影响优先级排序
   - 合併检查相同需求方面的近乎重复
   - 如果 >5 个低影响边缘情况，创建一个项目："需求中是否解决了边缘情况 X、Y、Z？"[覆盖率]

   **🚫 绝对禁止** - 这些使其成为实现测试，而不是需求测试：
   - ❌ 以"验证"、"测试"、"确认"、"检查" + 实现行为开头的任何项目
   - ❌ 对代码执行、用户行动、系统行为的引用
   - ❌ "正确显示"、"正常工作"、"按预期功能"
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 程序
   - ❌ 实现细节（框架、API、算法）

   **✅ 必需模式** - 这些测试需求质量：
   - ✅ "是否为[场景]定义/指定/记录了[需求类型]？"
   - ✅ "是否用具体标准量化/阐明了[模糊术语]？"
   - ✅ "[部分 A]和[部分 B]之间的需求是否一致？"
   - ✅ "能否客观衡量/验证[需求]？"
   - ✅ "需求中是否解决了[边缘情况/场景]？"
   - ✅ "规格是否定义了[缺失方面]？"

6. **结构引用**：使用 `templates/checklist-template.md` 中的规格模板生成标题、元部分、类别标题和 ID 格式的清单。如果模板不可用，请使用：H1 标题、目的/创建元行、`##` 包含 `- [ ] CHK### <需求项目>` 行且从 CHK001 开始全局递增 ID 的类别部分。

7. **报告**：输出生成的清单的完整路径、项目数，并提醒用户每次运行都会创建新文件。总结：
   - 选定的重点领域
   - 深度级别
   - 参与者/时间
   - 合并的任何显式用户指定的必须包含项目

**重要**：每个 `/speckit.checklist` 命令调用都使用基于领域的简短、描述性名称创建清单文件，除非文件已存在。这允许：

- 不同类型的多个清单（例如，`ux.md`、`test.md`、`security.md`）
- 指示清单目的的简单、易记的文件名
- 在 `checklists/` 文件夹中轻松识别和导航

为避免混乱，请使用描述性类型，并在完成后清理过时的清单。

## 清单类型和示例项目

**UX 需求质量：** `ux.md`

示例项目（测试需求，不是实现）：

- "是否用可衡量标准定义了视觉层次需求？"[清晰性, Spec §FR-1]
- "UI 元素的数量和定位是否明确指定？"[完整性, Spec §FR-1]
- "交互状态需求（悬停、焦点、活动）是否一致定义？"[一致性]
- "是否为所有交互元素指定了可访问性需求？"[覆盖率, Gap]
- "徽标图像加载失败时是否定义了回退行为？"[边缘情况, Gap]
- "能否客观衡量'显著显示'？"[可衡量性, Spec §FR-4]

**API 需求质量：** `api.md`

示例项目：

- "是否为所有失败场景指定了错误响应格式？"[完整性]
- "是否用特定阈值量化了速率限制需求？"[清晰性]
- "所有端点的认证需求是否一致？"[一致性]
- "是否为外部依赖项定义了重试/超时需求？"[覆盖率, Gap]
- "需求中是否记录了版本控制策略？"[Gap]

**性能需求质量：** `performance.md`

示例项目：

- "是否用具体指标量化了性能需求？"[清晰性]
- "是否为所有关键用户旅程定义了性能目标？"[覆盖率]
- "是否为不同负载条件指定了性能需求？"[完整性]
- "能否客观衡量性能需求？"[可衡量性]
- "是否为高负载场景定义了降级需求？"[边缘情况, Gap]

**安全需求质量：** `security.md`

示例项目：

- "是否为所有受保护资源指定了认证需求？"[覆盖率]
- "是否为敏感信息定义了数据保护需求？"[完整性]
- "是否记录了威胁模型且需求与之对齐？"[可追溯性]
- "安全需求是否符合合规义务？"[一致性]
- "是否为安全失败/违规响应定义了需求？"[Gap, 异常流程]

## 反例：不要做什么

**❌ 错误 - 这些测试实现，不是需求：**

```markdown
- [ ] CHK001 - 验证着陆页显示 3 个剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面上的悬停状态是否正确工作 [Spec §FR-003]
- [ ] CHK003 - 确认徽标点击导航到主页 [Spec §FR-010]
- [ ] CHK004 - 检查相关剧集部分显示 3-5 个项目 [Spec §FR-005]
```

**✅ 正确 - 这些测试需求质量：**

```markdown
- [ ] CHK001 - 特色剧集的数量和布局是否明确指定？"[完整性, Spec §FR-001]
- [ ] CHK002 - 所有交互元素的悬停状态需求是否一致定义？"[一致性, Spec §FR-003]
- [ ] CHK003 - 所有可点击品牌元素的导航需求是否清晰？"[清晰性, Spec §FR-010]
- [ ] CHK004 - 相关剧集的选择标准是否记录？"[Gap, Spec §FR-005]
- [ ] CHK005 - 异步剧集数据的加载状态是否定义？"[Gap]
- [ ] CHK006 - 能否客观衡量"视觉层次"需求？"[可衡量性, Spec §FR-001]
```

**关键区别：**

- 错误：测试系统是否正常工作
- 正确：测试需求是否编写正确
- 错误：行为验证
- 正确：需求质量验证
- 错误："它做 X 吗？"
- 正确："X 明确指定了吗？"
